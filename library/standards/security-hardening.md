# Security Hardening

## Objective

Integrate security as a first-class constraint in the code-writing flow (Phase D). Every suggestion generated by AI or written by humans must pass through this set of rules before being considered acceptable. Security is not a final phase — it is a continuous property of the code.

---

## Non-Negotiable Principles

1. **Zero Trust**: Never trust any input, token, header, or external result without explicit validation — regardless of origin.
2. **Least Agency**: Request only the strictly necessary permissions. No long-lived credentials. No god-mode access.
3. **Parameterize Always**: Every database query uses bound parameters. String concatenation in SQL is prohibited without exception.
4. **Secrets Out of Code**: Secrets live in environment variables or dedicated vaults. Never in source code, logs, or comments.
5. **Fail Closed**: On authorization or validation errors, block by default. Never allow by default on fallback paths.
6. **OWASP Baseline**: OWASP Top 10:2025 mitigations are the minimum floor — not the ceiling — of acceptable security.

---

## 1. Zero Trust

No request, internal service, JWT token, or microservice call is implicitly trusted. Every system boundary revalidates authentication, authorization, and data integrity.

### Always: Zero Trust

- Validate authentication and authorization at every layer (Controller, Service, Repository).
- Revalidate tokens on every request — not only at login.
- Treat headers, query params, body, and path params as untrusted input.
- Use mutual TLS (mTLS) for service-to-service communication in production.
- Apply rate limiting and circuit breakers to all external calls.

### Never: Zero Trust

- Never assume a valid token implies authorization for the requested resource.
- Never propagate security contexts via unsigned headers (e.g., `X-User-Id`).
- Never trust inputs validated by an upstream service without revalidating locally.
- Never disable SSL certificate verification (`rejectUnauthorized: false`) in production.

### Bad vs Good: Zero Trust

```typescript
// BAD — trusts the propagated header without revalidation
app.get('/orders/:id', (req, res) => {
  const userId = req.headers['x-user-id']; // untrusted, unverified
  return orderService.getOrder(req.params.id, userId);
});

// GOOD — extracts identity only from the signed token
app.get('/orders/:id', authenticate, async (req, res) => {
  const userId = req.user.id; // extracted from JWT verified by middleware
  const order = await orderService.getOrder(req.params.id, userId);
  if (!order || order.ownerId !== userId) {
    return res.status(403).json({ error: 'Forbidden' });
  }
  return res.json(order);
});
```

## 2. Least Agency

Code and AI agents must operate with the minimum set of permissions needed for the task at hand. Long-lived credentials and broad scopes are vectors of compromise.

### Always: Least Agency

- Define granular permission scopes per operation (read vs. write vs. admin).
- Use short-lived credentials with automatic rotation (e.g., AWS STS, Vault dynamic secrets).
- Revoke credentials immediately when the usage context ends.
- For AI agents: explicitly delimit which tools and resources the agent may access per task.
- Store secrets in dedicated vaults (e.g., AWS Secrets Manager, HashiCorp Vault, Azure Key Vault).

### Never: Least Agency

- Never use a single `admin` or `root` credential for multiple services.
- Never generate access tokens with broader scopes than needed for the current operation.
- Never hardcode credentials, tokens, API keys, or passwords in source code.
- Never log credentials, tokens, or sensitive data — not even in `debug` mode.
- Never pass secrets via command-line arguments (visible in `ps aux`).

### Bad vs Good: Least Agency

```typescript
// BAD — hardcoded credential with full access
const db = new DatabaseClient({
  host: 'prod-db.internal',
  user: 'admin',
  password: 'SuperSecret123!', // hardcoded, admin scope
});

// GOOD — environment credential, minimum scope, no hardcode
const db = new DatabaseClient({
  host: process.env.DB_HOST,
  user: process.env.DB_USER,
  password: process.env.DB_PASSWORD, // injected via env or secret manager
  // database user has permission only for SELECT/INSERT on the required schema
});
```

```typescript
// BAD — long-lived token stored inline
const apiClient = new ExternalService({
  apiKey: 'sk-live-abc123def456ghi789', // hardcoded, never expires
});

// GOOD — token read from environment variable at runtime
const apiClient = new ExternalService({
  apiKey: process.env.EXTERNAL_API_KEY, // rotated via secret manager
});
```

## 3. Input Sanitization — OWASP A05:2025 Injection

Every input that crosses a system boundary (HTTP, CLI, queue message, file) is untrusted until proven otherwise. Injection is the fifth most critical category in the OWASP Top 10:2025 and encompasses SQL, NoSQL, LDAP, OS Command, and Expression Language injection.

**Absolute Rule**: No external input is ever directly interpolated into a query, command, or executable expression. Always use bound parameters (parameterized queries / prepared statements).

### Always: Input Sanitization

- Use prepared statements or parameterized query builders for every database interaction.
- Validate and normalize inputs in DTOs at the application boundary — before any business logic.
- Use allowlists — never blocklists — to validate expected formats.
- Encode outputs in the correct context: HTML encode for HTML, JS encode for JavaScript, URL encode for URLs.
- Validate Content-Type, maximum size, and structure of every upload payload.

### Never: Input Sanitization

- **Never build SQL queries by string concatenation with user input** — no exceptions, no "special cases".
- Never use `eval()`, `Function()`, `exec()`, or equivalents with external data.
- Never trust sanitization performed only on the frontend — the backend revalidates everything.
- Never use permissive RegEx to validate emails, URLs, or identifiers — use dedicated parsers.
- Never return database error messages directly to the client (leaks schema and internal logic).

### Bad vs Good: SQL Injection (A05:2025)

```typescript
// BAD — direct concatenation: guaranteed SQL Injection
async function getUserByEmail(email: string) {
  const query = `SELECT * FROM users WHERE email = '${email}'`;
  // input: "'; DROP TABLE users; --" => disaster
  return db.query(query);
}

// GOOD — prepared statement with bound parameter
async function getUserByEmail(email: string) {
  const query = 'SELECT * FROM users WHERE email = $1';
  return db.query(query, [email]); // driver escapes the value correctly
}
```

```typescript
// BAD — ORM misused with raw interpolation
async function searchProducts(term: string) {
  return Product.findAll({
    where: sequelize.literal(`name LIKE '%${term}%'`), // injection via literal()
  });
}

// GOOD — ORM with parameterized operators
async function searchProducts(term: string) {
  return Product.findAll({
    where: { name: { [Op.like]: `%${term}%` } }, // sequelize escapes the binding
  });
}
```

### Bad vs Good: XSS (A05:2025)

```typescript
// BAD — renders HTML with user content without encoding
function renderComment(comment: string): string {
  return `<div class="comment">${comment}</div>`;
  // input: "<script>document.cookie</script>" => XSS
}

// GOOD — encode before inserting into the DOM
import { escapeHtml } from './utils/escape';

function renderComment(comment: string): string {
  return `<div class="comment">${escapeHtml(comment)}</div>`;
}
```

## 4. Secrets Prevention — OWASP A04:2025 Cryptographic Failures

Segredos expostos no código-fonte, logs ou variáveis de ambiente não criptografadas são a causa raiz de inúmeras brechas de dados. O OWASP A04:2025 (Cryptographic Failures) abrange tanto a ausência de criptografia quanto o uso incorreto de credenciais e chaves.

### Always: Secrets Prevention

- Carregue segredos exclusivamente de variáveis de ambiente ou de um secret manager em runtime.
- Use `.env.example` without real values as documentation of the configuration contract.
- Configure pre-commit hooks (e.g., `gitleaks`, `trufflehog`) to block commits containing secrets.
- Add `.env` and credential files to `.gitignore` globally.
- Run secret scans in CI/CD (e.g., `gitleaks detect` in the pipeline).
- Rotate compromised credentials immediately and invalidate previous versions.

### Never: Secrets Prevention

- **Never hardcode API keys, tokens, passwords, certificates, or any secret in source code.**
- Never commit `.env`, `*.pem`, `*.key`, `*credentials*`, or `config/secrets.*` files.
- Never log secret values — not even as `console.log('token:', token)` in debug.
- Never pass secrets in URLs (query params end up in server logs and browser history).
- Never store secrets in code comments, even "temporarily".
- Never use MD5 or SHA-1 hashes for sensitive data — use bcrypt, argon2, or SHA-256+ with salt.

### Bad vs Good: Secrets Prevention

```typescript
// BAD — secret hardcoded directly in code
const jwtSecret = 'my-super-secret-key-12345';
const token = jwt.sign(payload, jwtSecret);

// GOOD — secret injected via environment with presence validation
const jwtSecret = process.env.JWT_SECRET;
if (!jwtSecret) {
  throw new Error('JWT_SECRET environment variable is required');
}
const token = jwt.sign(payload, jwtSecret, { expiresIn: '1h' });
```

```typescript
// BAD — URL with credential exposed in logs
const response = await fetch(
  `https://api.service.com/data?api_key=sk-live-abc123&user=${userId}`
);

// GOOD — credential in Authorization header, never in the URL
const response = await fetch(`https://api.service.com/data?user=${userId}`, {
  headers: {
    Authorization: `Bearer ${process.env.SERVICE_API_KEY}`,
  },
});
```

## 5. OWASP Top 10:2025 — Mitigation Map

| # | Category | Primary Mitigation |
| --- | --- | --- |
| A01 | Broken Access Control | RBAC/ABAC per resource; check at every layer; deny-by-default |
| A02 | Security Misconfiguration | Infrastructure-as-Code with linting; remove debug endpoints in prod; security headers |
| A03 | Software Supply Chain Failures | Pin dependency versions; SBOM; vulnerability scan in CI (e.g., `npm audit`) |
| A04 | Cryptographic Failures | TLS 1.2+ mandatory; bcrypt/argon2 for passwords; no MD5/SHA-1 for sensitive data |
| A05 | Injection | Prepared statements; parameterized ORMs; DTO validation; never `eval()` |
| A06 | Insecure Design | Threat modeling at design phase; fail-closed principle; abuse cases in user stories |
| A07 | Authentication Failures | MFA mandatory; short-lived tokens; lockout after N failed attempts |
| A08 | Software/Data Integrity Failures | Artifact signature verification; dependency integrity (lockfiles); no `--force` on deploys |
| A09 | Security Logging Failures | Log authentication, authorization, and validation error events; never log sensitive data |
| A10 | Mishandling of Exceptional Conditions | Explicit handling of all errors; generic messages to clients; stack traces only in internal logs |

## 6. Broken Access Control — OWASP A01:2025

Broken access control is the most prevalent OWASP 2025 category. AI must not suggest code that assumes authentication implies authorization.

### Always: Access Control

- Implement resource-based authorization (ABAC) — not only role-based (RBAC).
- Verify that the requested resource belongs to the authenticated user before returning it.
- Use opaque internal IDs (UUID) instead of predictable sequential IDs in public endpoints.
- Apply the deny-by-default principle: if a rule does not grant access, block.

### Never: Access Control

- Never trust that `userId` in the URL corresponds to the authenticated user without cross-checking.
- Never expose administrative endpoints without additional authentication (e.g., `/admin`, `/actuator`).
- Never use predictable incremental IDs in private resources (IDOR — Insecure Direct Object Reference).

### Bad vs Good: Access Control

```typescript
// BAD — IDOR: any authenticated user can access any invoice
app.get('/invoices/:id', authenticate, async (req, res) => {
  const invoice = await Invoice.findById(req.params.id); // no ownership check
  return res.json(invoice);
});

// GOOD — verifies resource ownership before returning
app.get('/invoices/:id', authenticate, async (req, res) => {
  const invoice = await Invoice.findOne({
    where: { id: req.params.id, ownerId: req.user.id }, // scoped by owner
  });
  if (!invoice) return res.status(404).json({ error: 'Not found' });
  return res.json(invoice);
});
```

## Security Review Checklist

Before any PR is approved, confirm:

- [ ] No SQL query is built by string concatenation with external input
- [ ] No secret, token, password, or key is hardcoded in source code or tests
- [ ] All external input is validated in DTOs before entering business logic
- [ ] Credentials are loaded exclusively from environment variables or a secret manager
- [ ] Access control verifies resource ownership, not only authentication
- [ ] Internal errors are not returned directly to the client
- [ ] Dependencies are pinned with a lockfile and passed `npm audit` / equivalent
- [ ] Security headers are configured (CSP, HSTS, X-Frame-Options)
- [ ] Audit logs cover authentication, authorization, and validation errors
- [ ] No debug or admin endpoint is exposed without additional authentication in production

---

## Enforcement

**AI (GitHub Copilot / agents):**

- Refuse to generate code that concatenates external input into SQL queries — always suggest prepared statements.
- Refuse to hardcode any value that resembles a credential — replace with `process.env.VARIABLE_NAME`.
- When reviewing code, automatically flag: SQL concatenation, inline secrets, `eval()` with external data, disabled SSL.
- In ORM suggestions, always use parameterized operators — never `sequelize.literal()` or equivalents with user variables.

**CI/CD Pipeline:**

- `gitleaks detect` — blocks push with detected secrets.
- `npm audit --audit-level=high` — fails the build on high-severity vulnerabilities.
- SAST (e.g., Semgrep with OWASP ruleset) — mandatory static analysis on PRs.
- Dependency review — blocks introduction of packages with known CVEs.

**Code Review:**

- Any occurrence of template literals in database queries is an immediate blocking reason.
- Any string that resembles an API key, JWT token, or password is an immediate blocking reason.
- Reviewers must consult this document before approving code that interacts with databases, authentication, or external services.
